# Red Team: MTFP21 Full Forward Pass (2026-04-01)

## What we're claiming

A 2-billion-parameter language model (BitNet b1.58-2B-4T) produces coherent text with the attention and FFN blocks running in MTFP21 arithmetic. No float32 in the core compute. The int8 at matmul interfaces is a wire format for the sign_epi8 SIMD bus. RoPE uses precomputed CONST tables. Softmax uses mtfp21_exp.

## Is the generation genuine?

**YES, with caveats.**

The model produces coherent, factually accurate text across 5 diverse prompts (factual recall, physics, fiction, computer science, philosophy). No loops, no garbage, good lexical diversity.

**Caveat 1:** The text is DIFFERENT from baseline, not identical. The MTFP21 path produces different logit distributions, which means different sampling outcomes. "Different but coherent" is expected — the question is whether the logit distributions are equivalent in quality, not identical in value.

**Caveat 2:** We haven't run PPL on the full pipeline yet (attention + FFN both MTFP21). The FFN-only MTFP21 path matched baseline PPL (17.887 vs 17.873). The attention MTFP21 path hasn't been PPL-validated yet. The generation tests suggest quality is maintained, but PPL is the rigorous measure.

**Caveat 3:** Speed is 1.3 tok/s (vs 22 tok/s baseline). This is because MTFP21 per-element operations are scalar int64 loops. The attention is O(n² × heads) per token with scalar MTFP21 dot products. This is a performance problem, not a correctness problem — but it means we can't run extensive PPL benchmarks quickly.

## Is the MTFP21 path actually running?

**YES.** Verified by:
- `shirley: MTFP21 FFN active (layer 0, N tokens)` banner in stderr
- `shirley: MTFP21 attention active (layer 0, N tokens, kv_pos=N)` banner
- All 30 layers × 2 custom ops = 60 Shirley compute calls per forward pass
- The graph node count dropped from 1116 (float ggml) to 246 (Shirley custom ops)

## What's still float32?

1. **Embedding lookup** (once per token) — model.tok_embd is float32
2. **attn_norm** (per layer) — ggml RMSNorm, float32
3. **attn_sub_norm + wo matmul + wo_scale** (per layer) — between attention and FFN custom ops
4. **Output norm + LM head** (once) — float32 matmul with tied embedding weights
5. **The matmul interface** — MTFP21 values converted to float, quantized to int8, matmul produces float (int32 in float), then back to MTFP21

**Item 5 is the most significant.** Inside `mtfp21_pack_for_matmul`, we call `mtfp21_to_float` then quantize to int8. Inside `matmul_raw_to_mtfp21`, we call `mtfp21_from_float(raw[i])`. These float conversions are at the matmul boundary, not in the inter-op flow. But they're there.

**Honest assessment:** The core compute (between matmuls) is genuinely MTFP21. The matmul boundary uses float as a format conversion intermediary. This is analogous to converting between int8 and float for the SIMD bus — it's a wire format issue, not a precision issue. But calling this "zero float32" would be inaccurate. The float32 is transient (used only during the pack/unpack at each matmul call), not persistent (no float32 values are stored or passed between operations).

## Is the KV cache geometric?

**Partially.** The KV cache stores MTFP21 values converted to float (`mtfp21_to_float` on write, `mtfp21_from_float` on read). This means the cache is float32, not MTFP21-native. The geometric interpretation (exponents as coordinates) is correct in principle but not yet in implementation — the exponent information is lost when we convert to float for storage.

**To make it genuinely geometric:** Store `mtfp21_t` structs directly in the cache. This preserves the exponent coordinates. The dot product between Q and cached K would then be MTFP21 dot products with geometric proximity computed in the native coordinate system.

## Is the RoPE truly CONST?

**YES.** The sin/cos tables are computed once at model load using `sinf`/`cosf` (transcendentals consumed at load time). At inference time, the RoPE computation is:
```
q_rot = q * cos_table[pos] + q_swap * sin_table[pos]
```
This is MUL + ADD — two CPU-native primes. No runtime transcendentals. The CONST prime is correctly applied.

**However:** The cos/sin values in the tables are stored as float32, not MTFP21. When the RoPE function applies them, it calls `mtfp21_from_float(cos_table[i])` per element per head per token. This is a runtime float→MTFP21 conversion that could be eliminated by storing the tables as MTFP21 at load time.

## Is the softmax using the EXP prime correctly?

**YES, with a precision question.** `mtfp21_exp()` uses a 256-entry LUT + linear interpolation. Max error is 9.50e-06. For softmax on attention scores (range [-30, 0]), this precision is sufficient — the attention weights are dominated by the top few scores, and 9.5e-06 error on the tail is invisible.

**But:** The `mtfp21_exp` implementation uses float for range reduction (computing k and the LUT index). The LUT values themselves are MTFP21, and the output is MTFP21, but the range reduction step touches float. This is noted in the code as a TODO.

## What would break this?

1. **Long sequences.** The KV cache grows linearly. The attention is O(n²). At 1.3 tok/s, a 512-token sequence takes ~6 minutes. The MTFP21 attention dot product is the bottleneck — scalar loops over all cached positions × all heads × head_dim.

2. **The wo projection is still float.** Between the attention custom op output and the FFN custom op input, the data flows through: attention output (float32) → attn_sub_norm (ggml float RMSNorm) → wo matmul (ggml ternary, float post-processing) → wo_scale (ggml float) → residual add → FFN input. This is a float32 gap in the middle of the pipeline. Each layer crosses this gap.

3. **The attn_norm is still float.** The very first operation on each layer's input is attn_norm — a float32 ggml RMSNorm. This means the MTFP21 FFN output gets converted to float at the layer boundary, then the float gets converted back to MTFP21 by the attention custom op's input conversion.

4. **Cumulative precision from float↔MTFP21 conversions.** Each layer does: MTFP21→float (attention output) → float operations (sub_norm, wo, scale, residual, attn_norm) → float→MTFP21 (attention input for next layer). That's 30 round-trips through float32. Each conversion has ~1e-7 relative error. Over 30 layers, this could accumulate.

## Recommendations

1. **Run PPL on the full pipeline** (both attention + FFN in MTFP21). If PPL matches baseline, the float boundary crossings aren't hurting.

2. **Fold attn_sub_norm + wo + wo_scale into the attention custom op.** This eliminates the float gap between attention and FFN. The wo matmul is ternary — it belongs in Shirley.

3. **Fold attn_norm into the FFN custom op or make it a separate Shirley op.** This eliminates the float gap at layer boundaries.

4. **Store RoPE tables as MTFP21** instead of float. Eliminates per-element float→MTFP21 conversion at runtime.

5. **Store KV cache as MTFP21** instead of float. Makes the geometric interpretation real, not just conceptual.

6. **Profile the MTFP21 attention dot product.** Identify whether vectorization (AVX2 int32 lanes for mantissa operations) would make the speed practical.

## Verdict

The claim "MTFP21 attention and FFN producing coherent text" is **genuine**. The model works. The architecture is correct. The inter-op precision (MTFP21 between matmuls) is validated by the FFN PPL result.

The claim "zero float32" is **aspirational, not yet achieved**. Float32 exists at: matmul pack/unpack boundaries, KV cache storage, RoPE table storage, and the three ggml ops between attention and FFN (attn_sub_norm, wo, wo_scale). Closing these gaps is Phase 6 work — straightforward engineering following the established pattern.

The geometric insight (exponents as coordinates) is **correct in principle, partially implemented**. The KV cache stores float, not MTFP21. Making it genuinely geometric requires storing mtfp21_t structs directly.
