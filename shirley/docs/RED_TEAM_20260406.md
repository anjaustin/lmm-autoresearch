# Red Team: Multi-Threaded Attention + Barriers (2026-04-06)

## Changes Under Review

1. Multi-threaded QKV+wo matmuls in attention (partitioned by rows)
2. Head-parallel attention body (REVERTED -- hangs at seq_len ~22)
3. Adaptive barrier (shirley_barrier.h) with _mm_pause()
4. AVX2 int32 multiply for FFN trivials (int16 x int8 -> int32 -> shift -> int16)
5. Monotonic phase counters (attempted, reverted to phase-reset pattern)

## Findings

### CRITICAL: mullo_epi16 overflow (FIXED)

`_mm256_mullo_epi16` keeps only the low 16 bits of the int16 x int16 product. The FFN trivials compute gate^2 (max 6400) x up (max 80) = 512,000. This exceeds int16 range (32,767). The low 16 bits of 512,000 are garbage. Corrupted values propagate through sub_norm and down matmul, eventually producing degenerate logits that cause the sampler to loop.

**Fix:** Widen to int32 before multiply. `_mm256_cvtepi16_epi32` + `_mm256_cvtepi8_epi32` + `_mm256_mullo_epi32` + `_mm256_srai_epi32(7)` + `_mm_packs_epi32`. Process 8 elements per iteration (128-bit pack output).

### CRITICAL: Head-parallel attention body hangs (REVERTED, UNRESOLVED)

Threading Q@K^T + softmax + attn@V across 20 heads (5 per thread on 4 threads) hangs deterministically at sequence position ~22-23. Works at 1 or 2 threads.

Investigation ruled out:
- Data race in output (head slices are non-overlapping)
- Barrier pattern bug (same pattern works for matmuls)
- AVX trivials corruption (scalar path also hangs)
- Phase cycling race (same behavior with monotonic and reset phases)

Unresolved. The hang is not a deadlock (threads at 396% CPU). Likely a livelock or numerical issue that causes the sampler to spin. Needs GDB or fprintf instrumentation at the exact failure point.

### HIGH: sched_yield() unusable on loaded systems

Replacing spin-waits with sched_yield() causes 1-10ms reschedule delays per yield call when system load > 2. With 300+ barrier waits per token, this makes inference 100x slower or causes timeouts. sched_yield() is fundamentally wrong for sub-ms synchronization.

### MEDIUM: Phase overflow at 10M tokens

Monotonic phase counter: int32 overflows after ~10M tokens. At 20 tok/s = ~142 hours. Not a concern for interactive inference. Server deployments should reset KV cache (and thus phase counters) between sessions.

### MEDIUM: Barrier cache contention

300 atomic spin-wait cycles per token. Each barrier involves 4 threads writing to the same cache line (mt_phase or mt_threads_done). L1 invalidation traffic scales with thread count. _mm_pause() reduces pipeline stalls but doesn't eliminate the MESI invalidation traffic. Futex-based barriers would eliminate this entirely but have the phase-cycling deadlock (see above).

### LOW: Dead memory allocations

KV cache exp arrays (k_cache_exp, v_cache_exp) are allocated at init but never used -- the cache stores float via reinterpreted int32 arrays. ~2 x max_seq_len x kv_dim bytes of wasted heap.

FFN header additions (mt_gate_max, mt_up_max, mt_threads_done2) are allocated but unused after trivials threading was reverted.

### COSMETIC: Profile fields for removed phases

attn_qk_dot and attn_softmax are still in the profile struct but always zero (the body runs as a single SP_LAP(attn_av) block). Harmless but misleading in profile output.

## Verified Correct

- QKV matmul row partitioning: non-overlapping, each thread writes to its own slice of mt_q_f/mt_k_f/mt_v_f
- wo matmul row partitioning: same pattern, writes to mt_wo_f
- Phase gate prevents threads from reaching next barrier before thread 0 resets counter
- End-of-token sync: thread 0 sets phase=0, non-zero threads spin until they see 0
- _mm_pause() in all spin loops: correct x86 usage, reduces power and prevents speculative load stalls
- int32 AVX multiply: widened before multiply, no overflow possible (max 512,000 < INT32_MAX)
- packs_epi32(lo128, hi128): correct element ordering for sequential 8-element output
