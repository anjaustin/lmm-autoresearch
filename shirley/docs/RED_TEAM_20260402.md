# Red Team: Adaptive-Width MTFP Pipeline (2026-04-02)

## What we're claiming

A 2-billion-parameter language model runs with adaptive-width MTFP arithmetic across the entire core computation. All 210 ternary matmuls use `sign_epi16` with int16 mantissas — zero float conversion in the matmul path. Between-matmul operations (ReLU-squared, element-wise multiply, RMSNorm, softmax, RoPE) run in MTFP21. The KV cache stores native MTFP21. RoPE tables are precomputed MTFP21. The only float32 remaining is at the model boundaries (embedding lookup, LM head matmul).

## Is the generation genuine?

**YES.** Tested across 5+ diverse prompts:
- Factual: "The capital of France is Paris, and the capital of Germany is Berlin."
- Scientific: "the position and momentum of a particle cannot both be precisely known at the same time"
- Creative: "there lived a young prince named Arin"
- Technical: "machine learning does not require explicit programming of the algorithm"

No loops, no garbage, good lexical diversity, factual accuracy maintained.

## Is sign_epi16 actually running?

**YES.** Verified by:
- `shirley: MTFP16 attention active (layer 0, sign_epi16 QKV+wo, zero float matmul)` banner
- `shirley: MTFP16 FFN active (layer 0, sign_epi16 matmul, zero float)` banner
- The `shirley_mtfp16_matmul.h` kernel was validated standalone: 6/6 tests, 1000-trial statistical validation (avg error 8e-05, max 0.5%)
- The ggml_gemv_i2_i8_s extern declaration is no longer in shirley_ffn.cpp — the old float path is removed

## Is the float truly gone from the matmul path?

**YES for the matmul wire.** The path is:
```
MTFP21 → mtfp16_block_align (trit-shift, pure integer)
→ int16 mantissas + block exponent
→ sign_epi16 × 2-bit ternary weights
→ int32 accumulator
→ MTFP21 (mantissa = int32 result, exponent = block_exp, normalize)
```

No `mtfp21_to_float`, no `mtfp21_from_float`, no `quantize_row_i8_s`, no float scaling. Verified by code inspection.

**BUT: float still exists at these points:**
1. `mtfp21_from_float(input[i])` — converting ggml's float32 layer input to MTFP21 (line ~145 in both attn and ffn compute). This happens once per token per layer at the top of each custom op.
2. `mtfp21_to_float(result)` — converting MTFP21 back to float32 for ggml output (line ~230 in attn, ~175 in ffn). This happens once per token per layer at the bottom of each custom op.
3. `mtfp21_from_float(gamma[i])` — converting float32 norm gamma weights to MTFP21. This happens per-element per RMSNorm call. Could be precomputed.
4. `mtfp21_from_float(p->kq_scale)` — converting the attention scale factor. One call per attention head per token.

**Honest assessment:** The matmul path itself is zero-float. But the conversion at the custom op boundaries (ggml float ↔ MTFP21) is not eliminated — it's the interface between ggml's float tensor system and Shirley's MTFP domain. These conversions happen 2× per layer (in + out) × 30 layers = 60 conversion points per token.

## Is the KV cache genuinely geometric?

**PARTIALLY.** The cache stores mantissa (int32) and exponent (int8) in parallel arrays. The exponent — the geometric coordinate — IS preserved. Each cached K and V entry retains its position in base-3 space.

**But:** The attention dot product Q@K^T reads from the cache and constructs mtfp21_t structs on the fly: `{p->k_cache_mant[idx], p->k_cache_exp[idx]}`. This is a struct construction, not a geometric operation. The dot product itself is scalar MTFP21 — it doesn't exploit the geometric structure of the cache (e.g., pruning distant landmarks, hierarchical search by exponent similarity). The geometric interpretation is preserved in storage but not yet exploited in computation.

## Is the RoPE genuinely CONST?

**YES.** Sin/cos tables are computed at model load via `sinf`/`cosf` → `mtfp21_from_float` → stored as mantissa + exponent arrays. At runtime, RoPE reads MTFP21 directly: `{cos_mant[i], cos_exp[i]}`. Zero float conversion at inference time. The transcendentals are consumed at load and frozen as CONST prime values.

## What's the precision situation?

**MTFP16 matmul wire:** 10-trit mantissa, 15.8 bits. Validated at avg 8e-05 error, max 0.5% across 1000 random vectors. Exceeds float16's 11-bit mantissa. Post-normalization data loses 0 values to block exponent alignment.

**MTFP21 between-matmul ops:** 16-trit mantissa, 25.4 bits. Exceeds float32's 24 bits. This is where the precision lives — ReLU, square, multiply, softmax all operate at full MTFP21.

**Concern:** The block exponent alignment in `mtfp16_block_align` does a trit-shift loop per element. For vectors where exponents are tightly clustered (post-normalization), this is fast (0-2 shifts). For vectors with wide dynamic range (attention weights), the loop could shift many positions, losing mantissa trits. This hasn't been tested on attention-weight-scale vectors.

## What would break this?

1. **Long sequences with the MTFP21 attention.** The Q@K^T dot product is O(kv_len × head_dim) per head, with scalar MTFP21 ops. At 512 tokens × 20 heads × 128 dim × 30 layers, that's ~39 billion MTFP21 operations per prompt. At ~30ns per op, that's ~20 minutes per prompt. The performance cliff is real for long contexts.

2. **The block exponent alignment on non-normalized data.** RMSNorm output has tight exponent spread. But the QKV matmul output (before RoPE) might not — the weight scale varies by layer. If the matmul output has wide exponent spread, block alignment crushes outliers. This needs measurement.

3. **The gamma weight conversion.** Every RMSNorm call converts float gamma weights to MTFP21 per-element. With 5 norms per layer × 30 layers × 2560 elements, that's 384,000 `mtfp21_from_float` calls per token — each involving `log()` and `pow()`. These should be precomputed at model load.

4. **The ggml boundary conversions.** 60 float↔MTFP21 conversion points per token (2 per layer × 30 layers). Each involves n_embd=2560 elements of `mtfp21_from_float` or `mtfp21_to_float`. That's 153,600 conversion calls per token. Not negligible.

## Recommendations

1. **Precompute gamma weights as MTFP21 at model load.** Store mantissa + exponent arrays alongside the float gamma. Eliminates 384K `mtfp21_from_float` calls per token.

2. **Measure block exponent alignment quality on matmul output.** Check exponent spread of QKV/gate/up/down matmul outputs. If spread > 5 trits, the block alignment is losing precision.

3. **Profile the conversion overhead.** How much of the 2.81 tok/s is spent in `mtfp21_from_float` / `mtfp21_to_float` at the custom op boundaries? If significant, consider keeping MTFP21 state between custom ops instead of converting to/from ggml float.

4. **Document the adaptive-width MTFP family formally.** The MTFP16 spec exists but isn't in the MTFP21.md document alongside the full-width spec.

## Verdict

The claim "adaptive-width MTFP pipeline with zero-float matmul" is **genuine**. The matmul path (`sign_epi16` × 2-bit ternary) has no float conversion. Between-matmul ops run at MTFP21 full precision. KV cache preserves geometric coordinates. RoPE tables are frozen MTFP21.

The claim "zero float32 in the core pipeline" requires clarification: zero in the matmul wire, zero in the KV cache, zero in RoPE. But float32 persists at the ggml interface (custom op input/output conversion) and in the gamma weight conversion (should be precomputed). These are optimization targets, not architectural flaws.

The model produces coherent, factually accurate text. The architecture is correct. The remaining float is at boundaries that have clear remediation paths.
